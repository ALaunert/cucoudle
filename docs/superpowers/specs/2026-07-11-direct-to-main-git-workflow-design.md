# Cucoudle: direct-to-main Git-процесс хакатона

## Цель

Сократить время на интеграцию изменений во время хакатона, отказавшись от обязательных feature-веток и pull request для обычной разработки, сохранив защиту от потери чужих изменений.

## Основной процесс

- Разработка ведётся непосредственно в ветке `main`.
- Завершённый и проверенный инкремент коммитится и сразу отправляется в `origin/main`.
- Создание отдельной ветки, merge request или pull request не требуется, если пользователь явно не попросил об этом.
- Force-push и другие способы переписывания истории `main` запрещены.

## Отклонённый push

Если push отклонён из-за того, что удалённая `main` ушла вперёд:

1. выполнить `git pull --rebase origin main`;
2. при конфликтах самостоятельно изучить обе стороны и контекст;
3. разрешить каждый конфликт с сохранением совместимого смысла изменений;
4. добавить разрешённые файлы и продолжить `git rebase --continue`;
5. повторять до завершения rebase;
6. заново выполнить относящиеся к инкременту проверки;
7. повторить обычный push без `--force`.

Агент обращается к пользователю только в крайнем случае: когда варианты разрешения действительно взаимоисключающие, намерение невозможно восстановить из кода и документации либо продолжение создаёт существенный риск потери данных.

## Работа с незакоммиченными изменениями

Перед pull/rebase агент проверяет состояние рабочего дерева. Он не перезаписывает и не прячет автоматически неизвестные пользовательские изменения. Если изменения относятся к текущему инкременту, их следует проверить и закоммитить; если они чужие или их назначение неясно, необходимо сохранить их и выбрать безопасный способ продолжения.

## Документирование

Решение о direct-to-main процессе фиксируется в `docs/PROGRESS.md` и в актуальном описании процесса разработки в `docs/FINAL_IMPLEMENTATION.md`.

## Проверка

- `AGENTS.md` прямо разрешает push в `main`.
- Отдельные ветки и PR не требуются по умолчанию.
- Для non-fast-forward указан `git pull --rebase origin main`.
- Конфликты разрешаются агентом самостоятельно.
- Force-push явно запрещён.
- После rebase требуется повторная проверка перед push.
